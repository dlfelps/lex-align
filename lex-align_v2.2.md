

## LexAlign v2.2: Enterprise Governance Platform Specification

**Version:** 2.2
**Date:** April 26, 2024
**Status:** Final

### 1. Product Vision & Value Proposition

LexAlign is an enterprise governance platform that ensures all code generated by AI agents or developers is legally compliant, architecturally sound, and secure.

By centralizing policy enforcement, LexAlign moves governance from a reactive process to a proactive, collaborative planning tool for AI agents. It prevents compliance issues before they are committed, provides clear intelligence to security and legal teams, and accelerates development by steering agents toward a "golden path" of safe dependencies.

### 2. System Architecture

LexAlign v2.2 operates on a centralized client-server model designed for scalability, consistency, and performance.

*   **LexAlign Server (The Core Platform):** A single, authoritative service that hosts all policy, logic, and data. It is the single source of truth for all compliance decisions.
*   **LexAlign Client (The Thin Agent):** A lightweight, stateless tool that integrates with the AI agent's environment. It acts as both a proactive "Advisor" and a mandatory "Guardrail."

### 3. Core Principle: "Use First, Approve in Parallel"

The system is built on a principle of zero-friction enforcement. The agent's progress is never blocked by administrative processes.

**A new, unknown package that passes all automated, real-time checks (license, CVEs) is considered provisionally approved for immediate use.** The process of formally adding the package to the registry runs asynchronously and does not impede the agent's primary task.

**Workflow:**
1.  **Agent Inquires:** The agent uses `lex-align-client check` on a new package.
2.  **Server Grants Provisional Approval:** The server performs real-time checks. If they pass, it returns an immediate `ALLOW` verdict with a status of `PROVISIONALLY_ALLOWED`.
3.  **Agent Proceeds:** The agent uses this approval to immediately proceed with its coding task, adding the dependency and committing the code.
4.  **Agent Requests Approval (Asynchronously):** After completing its main task, the agent uses `lex-align-client request-approval` to submit the new package for formal inclusion in the registry. This action triggers a non-blocking, human-in-the-loop review process on the server side.

### 4. Client Workflow & Commands

The client provides a clear, purpose-driven set of commands for the AI agent.

#### 4.1. `lex-align-client init` (Setup)
*   **Purpose:** The one-time setup command for a new project.
*   **Actions:** Runs an interactive wizard to create the `.lexalign.toml` config file, provides instructions for secure API key handling (if in Organization Mode), and installs the pre-commit hook.

#### 4.2. `lex-align-client check` (The "Advisor")
*   **Purpose:** The primary, proactive tool for the AI agent. It is used during the planning phase to determine the compliance status of a potential dependency *before* code is written.
*   **Usage:** `lex-align-client check --package="<package-name>"`
*   **Output:** A structured JSON response to `stdout`, which the agent can parse to inform its plan.

#### 4.3. `lex-align-client request-approval` (The "Good Citizen")
*   **Purpose:** An asynchronous, non-blocking command used by the agent to request that a `PROVISIONALLY_ALLOWED` package be formally added to the enterprise registry.
*   **Usage:** `lex-align-client request-approval --package="<package-name>" --rationale="<why-it-was-needed>"`
*   **Action:** This sends the request to the server, which then automatically creates a pull request for the human review team. The agent does not wait for a response.

#### 4.4. The Pre-Commit Hook (The "Guardrail")
*   **Purpose:** A mandatory, non-negotiable enforcement layer that guarantees no non-compliant code is ever committed.
*   **Action:** Runs automatically on every `git commit`. It internally uses the same logic as the `check` command to validate all dependency changes staged for the commit.
*   **Behavior:**
    *   **On Success:** The commit proceeds silently.
    *   **On Failure:** The commit is **blocked**. A structured error message, identical to the `check` command's output, is printed to `stderr`, providing the agent with the necessary context to self-correct and attempt the commit again.

### 5. Deployment & Authentication Model

Deployment is via a `docker-compose.yml` file defining the `lexalign-server` and `redis` services. The server operates in one of two modes, controlled by an `AUTH_ENABLED` environment variable.

*   **Mode 1: Single-User Mode (Default: `AUTH_ENABLED=false`):** No authentication. The server is secured by binding to `localhost` only, making it perfect for frictionless local evaluation.
*   **Mode 2: Organization Mode (`AUTH_ENABLED=true`):** Requires API key authentication. The server can be exposed to the network. Keys are managed via a secure `lex-align-server admin` CLI.

### 6. Core Server Components

#### 6.1. The Policy Registry
A `registry.yml` file defining all package and license rules, managed via a Git repository.

#### 6.2. The API Server
Exposes RESTful endpoints. If `AUTH_ENABLED` is true, all endpoints require a valid Bearer token.
*   **`GET /api/v1/evaluate`**: Called by `lex-align-client check`. Returns a detailed JSON response, including statuses like `ALLOWED`, `DENIED`, or `PROVISIONALLY_ALLOWED`, and the `is_requestable` flag.
*   **`POST /api/v1/approval-requests`**: Called by `lex-align-client request-approval`. Triggers the automated pull request creation workflow.

#### 6.3. The Caching Layer (Redis)
Leverages Redis for high-performance caching of license and CVE data.

#### 6.4. The Audit Log & Database (SQLite)
Persists all decisions and API key hashes (if enabled) in a simple SQLite database.

#### 6.5. The Dashboards
Web interfaces for human administrators to manage the system.
*   **Security & Legal Dashboards:** For monitoring and review.
*   **Registry Workshop:** A UI for managing the `registry.yml` via the Git-based pull request workflow, serving as the destination for the AI's approval requests.

### Appendix: Evolutionary Migration from v1.x to v2.2

The transition from the original prototype is phased to ensure continuous value.

*   **Phase 1: Serverize & Dockerize.** Refactor the core logic into a server application, bundle it with Redis using Docker Compose, and establish the dual-mode authentication system.
*   **Phase 2: Implement the Thin Client.** Gut the original client, implementing the simplified `init`, `check`, and pre-commit hook logic that communicates with the server.
*   **Phase 3: Build the Approval Workflow.** Implement the `request-approval` command and the server-side logic to automatically generate pull requests from agent submissions.
*   **Phase 4: Build Intelligence Layers.** Develop the dashboards for human administrators to review the AI's requests and manage the overall system.
